FE. Favorite Events & Notification System

General Description:
The system shall provide a "Favorite Events" feature that allows users to save events they are interested in and access them later from a dedicated page. This feature helps users follow relevant events without searching for them repeatedly. In addition, the system shall monitor important changes related to favorite events, such as price decreases or newly available seats, and notify the user inside the application. The feature is intended to improve usability, support faster decision-making, and increase user engagement with the platform.


1. Navigation and Interface

FE.1: The system shall provide a "Favorite" button on the page of every event. This button allows the user to quickly add the event to their personal list of favorite events.

FE.2: When the user selects the Favorite button, the system shall store the selected event in the user's favorites list in the database.

FE.3: The system shall provide a dedicated "My Favorites" page where the user can view all events they have marked as favorites.

FE.4: The My Favorites page shall display the main information for each saved event, including at least the event title, date, location and current price.

FE.5: The system shall allow the user to remove an event from the favorites list by pressing a "Remove from Favorites" button.


2. Favorite Management Logic

FE.6: The system shall ensure that the same event cannot be added twice to the favorites list for the same user. If the user tries to add the same event again, the system shall ignore the request.

FE.7: The system shall store favorite events separately for each user, so every user has their own independent list of favorite events.

FE.8: The system shall only allow events that already exist in the database to be added to the favorites list.

FE.9: The system shall allow the user to view all favorite events at any time through the "My Favorites" page.


3. Notification Logic

FE.10: The system shall monitor changes related to the events saved in the user's favorites list.

FE.11: The system shall notify the user when the price of a favorite event decreases.

FE.12: The system shall notify the user when new seats become available for an event that was previously full.

FE.13: For seat availability, the system shall consider a relevant change when the number of available seats changes from 0 to a value greater than 0.

FE.14: The system shall create at most one notification for each relevant change, in order to avoid duplicate notifications for the same user and the same event state.

FE.15: Each notification shall include at least the event title, the type of change (price decrease or seats available), a short message and the date of creation.

FE.16: Notifications shall be displayed inside the application interface and shall also be stored in the database for each user so they can be viewed later.


4. Example Business Logic

This feature contains several simple but important business rules. The system must validate that an event exists before adding it to favorites, must prevent duplicates for the same user, and must keep favorite lists separate for different users. In addition, the system must compare the current state of an event with the previous known state in order to decide whether a notification should be created. For example, if the price changes from 50 to 40, the user should be notified; however, if the price remains 40, the system should not generate another identical notification. Similarly, if an event was full and seats later become available, the system should notify interested users once for that relevant change.
