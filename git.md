# Git + GitHub + Jira Workflow Guide

This document explains how we work with **Git, GitHub and Jira** during the project.
The goal is to keep our work organized, avoid conflicts, and make collaboration smooth.

---

# 1. General Workflow

The normal flow for every task looks like this:

```
Jira Task
   ↓
Move task → In Progress
   ↓
Branch automatically created
   ↓
Work locally
   ↓
Push commits
   ↓
Open PR
   ↓
Review (Rares)
   ↓
Merge to main
   ↓
Task automatically moves to Done
```

**Important rules**

* Never push directly to `main` (GitHub protection prevents this).
* Every branch corresponds to **one Jira task**.
* Every commit and PR must reference the **Jira task key**.
* Branches are **automatically created by Jira** when work begins.

---

# 2. Jira Workflow

Each task / feature / bug is tracked in **Jira**.

Workflow:

```
To Do → In Progress → In Review → Done
                ↑
          (if PR declined)
```

Meaning:

### To Do

The task exists but nobody is working on it yet.

### In Progress

Work started.
A branch is automatically created for the task.

### In Review

A Pull Request was opened and the code is being reviewed.

### Done

The Pull Request was merged into `main`.

---

# Automation Rules

Jira is configured with the following automations:

### 1️⃣ Create Branch on In Progress

When a task is moved `To Do → In Progress`, 
Jira automatically creates a branch in GitHub. (It may take a few seconds for the branch to get created)

This rule **does NOT run if the issue has the label** `no-branching`.

This allows documentation or planning tasks that don't require code.

---

### 2️⃣ PR Opens → Task moves to In Review

When a Pull Request is opened referencing the task Jra automatically moves `Task → In Review`

---

### 3️⃣ PR Declined → Task moves back to In Progress

If a PR is closed without merging:

```
Task → In Progress
```

So fixes can be applied.

---

### 4️⃣ PR Merged → Task moves to Done

When the PR is merged into `main`:

```
Task → Done
```

---

# 3. Branch Naming Convention

Branches are **automatically generated by Jira** when the issue moves to **In Progress**.

Branch format:

```
type/JIRA-KEY-short-description
```

Example:

```
feat/KAN-23-add-route-api
fix/KAN-45-login-validation
docs/KAN-12-update-readme
```

The branch name always contains the **Jira issue key** so GitHub and Jira stay linked.

---

## Branch Types

```
feat      → new feature
fix       → bug fix
refactor  → code improvement without behavior change
docs      → documentation changes
chore     → maintenance / config / small tasks
```

---

# 4. Commit Message Convention

Commit format:

```
type(JIRA-KEY): short description
```

Examples:

```
feat(KAN-18): add event filtering
fix(KAN-21): fix crash when event list is empty
refactor(KAN-11): simplify auth middleware
docs(KAN-7): update project setup instructions
chore(KAN-3): add build configuration
```

Guidelines:

* Keep messages short
* Use present tense
* Focus on **what changed**

Good example:

```
feat(KAN-22): add event creation endpoint
```

Please don't do:

```
stuff
update
fix things
```

# 5. Basic Git Commands

This section covers the commands we will use most often.

---

# Clone the repository

First time setup:

```bash
git clone <repo-url>
cd <repo>
```

---

# Update local repository

Always update your local `main` before starting work.

```bash
git switch main
git pull origin main
```

---

# Fetch newly created Jira branches

When a task moves to **In Progress**, Jira creates the branch automatically.

Fetch it locally:

```bash
git fetch --all
```

List remote branches:

```bash
git branch -r
```

Then create a local branch tracking the remote branch:

```bash
git switch -c branch-name origin/branch-name
```

Example:

```bash
git switch -c feat/KAN-23-add-route-api origin/feat/KAN-23-add-route-api
```

---

# Check branch status

```bash
git status
```

Shows changed files and staged files.

---

# Stage changes

Add specific files:

```bash
git add file_name
```

Add everything:

```bash
git add .
```

---

# Commit changes

```bash
git commit -m "feat(KAN-23): add route api endpoint"
```

---

# Push commits

```bash
git push
```

Since the branch already tracks the remote branch, this is enough.

---

# 6. Opening a Pull Request

After pushing your commits:

1. Go to the GitHub repository
2. Click **New Pull Request**
3. Base branch: `main`
4. Compare branch: your branch

PR title example:

```
feat(KAN-23): add route api endpoint
```

PR description example:

```
Jira: KAN-23

Summary:
Implemented the API endpoint for creating routes.

Changes:
- added route handler
- added validation
- added database insert
```

Once the PR is opened:

```
Task automatically moves → In Review
```

---

# 7. Keeping Your Branch Updated

If `main` changes while you are working:

Fetch updates:

```bash
git fetch origin
```

Rebase your branch:

```bash
git rebase origin/main
```
### IMPORTANT: 
If the main branch does not contain changes related to yours (same file, structure, etc.) don't rebase. 
It will create a mess in the history.

If conflicts appear:

1. Fix the files
2. Stage changes

```bash
git add .
```

3. Continue rebase

```bash
git rebase --continue
```

Then update the remote branch:

```bash
git push --force-with-lease
```

---

# 8. After PR is Merged

When the PR is merged:

```
Task automatically moves → Done
```

Update your repository:

```bash
git switch main
git pull origin main
```

Delete your local branch:

```bash
git branch -d branch-name
```

If Git complains that the branch is not fully merged (most probably because of squash merges):

```bash
git branch -D branch-name
```

---

# 9. Tasks Without Code

For tasks that don't require code (documentation, planning, diagrams):

Add the label:

```
no-branching
```

This prevents Jira from creating a branch automatically.

---

# 10. If Something Goes Wrong

Git mistakes happen.

Before panicking:

1. Ask for help in the team chat or dm

2. We'll fix it together.

Almost everything in Git can be recovered (with some magic).
